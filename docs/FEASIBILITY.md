## 2. 单节点算力门槛

### 2.1 最低配置（可参与计算）

| 硬件 | 最低要求 | 说明 |
|-----|---------|------|
| GPU | ≥8 GB VRAM（RTX 3070 / 4060 Ti 及以上） | 仅运行注意力层，约 3–5 层/节点 |
| CPU RAM | ≥64 GB DDR4/DDR5 | 承载 ~2 个 MoE 层的全量专家权重 |
| 存储 | ≥1 TB NVMe SSD（顺序读 ≥3 GB/s） | 权重分片 mmap，冷专家从磁盘页入 |
| 网络 | ≥1 Gbps 上行带宽，延迟 ≤20ms（至同区域节点） | 每 Token 约传输 14 KB TensorPacket |
| 系统 | Linux（推荐 Ubuntu 22.04）+ CUDA 12.x | KTransformers C++ 内核依赖 |

### 2.2 推荐配置（生产级参与）

| 硬件 | 推荐规格 | 说明 |
|-----|---------|------|
| GPU | RTX 4090 / 5070 Ti（24 GB VRAM） | 可承载 8–10 层注意力，减少节点总数 |
| CPU RAM | 128–256 GB DDR5 | 承载 5–10 个 MoE 层专家权重，命中率更高 |
| 存储 | 2–4 TB NVMe RAID（读 ≥7 GB/s） | 大幅减少专家换页延迟 |
| 网络 | ≥10 Gbps，延迟 ≤5ms（同城节点） | 流水线气泡最小化 |

### 2.3 各硬件配置下的层承载能力估算

```
GPU VRAM → 注意力层数：
  8 GB  → ~3 层（Attention 权重 ~400 MB/层，需留余量给激活值）
  16 GB → ~6 层
  24 GB → ~10 层

CPU RAM → MoE 层热专家容量（top-8 路由，256 专家）：
  64 GB  → ~2 个 MoE 层全量专家常驻
  128 GB → ~5 个 MoE 层全量专家常驻
  256 GB → ~11 个 MoE 层全量专家常驻
  （超出部分从 NVMe mmap 换页，延迟约 1–5 ms/次）
```

## 3. 地理微集群划分方案

### 3.1 划分原则

地理微集群的核心目标是**将节点间 RTT 控制在 MoE 专家分发的可接受延迟内**。

```
节点间 RTT 预算：
  单次 Token 推理中，MoE 分发产生 top-K=8 次专家调用
  若每次调用需跨节点通信，RTT 累积为推理延迟的主要来源

容忍上限：
  目标吞吐 2 tok/s → 每 Token 预算 500 ms
  注意力计算 ~50 ms → 剩余 ~450 ms 给 MoE 通信
  8 次专家调用，每次 RTT ≤ 50 ms → 节点间物理距离 ≤ ~2500 km
```

### 3.2 分级划分标准

| 集群级别 | RTT 要求 | 地理范围（估算） | 典型场景 |
|---------|---------|--------------|---------|
| **L0 本地节点** | <1 ms | 同一机器/局域网 | 家庭多 GPU 机器 |
| **L1 城市集群** | <5 ms | 同城市 / 同数据中心 | 同城朋友组队 |
| **L2 区域集群** | <20 ms | 同国 / 同大洲 | 国内用户组成的集群 |
| **L3 大陆际集群** | <80 ms | 跨洲际 | 仅作为备用节点，不参与热路径 |

> L3 节点（跨洲际）**不应出现在 MoE 专家分发的关键路径上**，仅可承担注意力层（通信量更小）或作为存储节点。

## 4. 最小可行集群规模

### 4.1 能跑完整 61 层推理的最小节点数

| 节点 GPU | RAM | 层数/节点 | 所需节点数 | 预计吞吐 |
|---------|-----|---------|---------|---------|
| RTX 3070 8 GB | 64 GB | 3 层 | ~21 节点 | ~0.5 tok/s |
| RTX 4090 24 GB | 128 GB | 10 层 | ~7 节点 | ~2 tok/s |
| RTX 5070 Ti 16 GB | 128 GB | 6 层 | ~11 节点 | ~1.5 tok/s |
| A100 80 GB | 512 GB | 61 层 | 1 节点 | ~15 tok/s（单机完整） |

> 以上吞吐估算基于 KTransformers 公开 benchmark（CPU offload 模式），实际性能受 NVMe 速度、节点间带宽影响显著。

### 4.2 带宽需求估算

```
每个 TensorPacket 大小：
  seq_len=1（单 Token 解码）× hidden_dim=7168 × float16 = 14 KB

节点间每秒传输量：
  2 tok/s × 14 KB/hop × 节点跳数 = 2 × 14 KB × 10 hops = 280 KB/s

流式传输（seq_len=512 prefill）：
  512 × 14 KB = 7 MB / hop → 需要 ~56 MB/s = ~450 Mbps（跨节点）

结论：
  单 Token 解码：1 Mbps 以上网络即可满足
  大批量 Prefill：需要 ≥1 Gbps 节点间带宽
```

## 5. 主要可行性风险与缓解

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| **MoE 专家换页延迟** | NVMe 读取热专家需 1–5 ms，累积 8 次则 8–40 ms | 共享专家常驻内存（已实现）；热专家 LRU 预测预加载 |
| **节点故障导致流水线中断** | 单节点挂掉整条推理链断开 | Orchestrator 重试 + DHT 动态替换（已实现框架） |
| **网络不稳定引发 RTT 抖动** | P99 延迟远高于 P50，影响用户体验 | 微批次并行、本地节点优先、超时截断 |
| **参与节点算力参差不齐** | 最慢节点成为整体瓶颈 | 按层数动态分配，慢节点承担更少层 |
| **模型权重分发** | 568 GB 权重如何分发给所有节点 | BT 协议分片分发 + 分层下载（只下本节点负责层的权重） |
| **冷启动问题** | 新节点无权重缓存，首次运行极慢 | 加入前预下载本节点权重分片（离线预热） |
| **梯度/模型正确性** | 分布式执行结果与单机执行是否一致 | 需要数值对齐测试（atol=1e-3），见 TESTING.md |

## 6. 与同类项目的对比

| 方案 | 硬件要求 | MoE 支持 | 地理 P2P | 开源 |
|-----|---------|---------|---------|------|
| **Astra（本项目）** | 8 GB GPU + 64 GB RAM | ✅ CPU offload | ✅ 设计目标 | ✅ Apache 2.0 |
| Petals | 8 GB GPU | ❌ 仅 dense 模型 | ✅ 已实现 | ✅ Apache 2.0 |
| KTransformers | 24 GB GPU + 128 GB RAM | ✅ CPU offload | ❌ 单机 | ✅ Apache 2.0 |
| vLLM | 40 GB+ GPU（多卡） | ✅ Tensor Parallel | ❌ 数据中心 | ✅ Apache 2.0 |
| llama.cpp | 仅 CPU 可行 | ❌ 不支持大规模 MoE | ❌ 单机 | ✅ MIT |

**Astra 的差异化定位：** 唯一将 Petals 的 P2P 网络与 KTransformers 的 MoE CPU offload 结合，面向**消费级显卡集群**运行**大规模 MoE 模型**的开源框架。
